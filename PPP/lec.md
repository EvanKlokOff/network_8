### WAN Encapsulation Protocols
В WAN Прежде чем отправиться, данные должны быть инкапсулированы в кадр. Для этого используются специальные протоколы инкапсуляции. Выбор такого протокола зависит от оборудования, а также типа WAN, который мы хотим создать

![alt text](image-2.png)

HDLC - тип инкапсуляции по умолчанию для соединения точка-точка, при использовании 2 цисочных устройств

PPP - обеспечивает соединение типа "роутер - роутер" и "хост - сеть" в асинхронном и синхронном каналах. Может работать с разными протоколами сетевого уровня

SLIP - стандартный протокол для последовательных соединений точка - точка в стеке tcp/ip

X.25 - Стандартный протокол. Предшественник frame relay. Определяет вариант подключения между DCE для удаленного доступа и связи компьютеров в сетях

Frame Relay -  коммутируемый протокол. Обрабатывает несколько виртуальных каналов.

ATM - использует передачу ячеек фиксированный длины для передачи данных / телефонии

Все технологии устарели


### HDLC

смотри структуру кадра

![alt text](image-3.png)

**читай слева направо**

Flag - используются для того, чтобы обозначать начало и конец кадра данных, т.к в канале связи отсутствуют механизмы маркировки

Далее идёт поле адреса, взятое из кадра канального уровня. Т.к канал точка- точка, то поле содержит 2 значения: направление передачи(от устройства к сети или из сети к устройству)

Control - 1 или 2 байта, его структура зависит от типа передаваемого кадра. Может быть информационные данные или управляющие данные

Поле Данных - передаются данные вышестоящих протоколов (с 7ого уровня)

Стандартный HDLC предполагает, что работаем в сетях с одним протоколом, поэтому не поддерживает определение типа протокола сетевого уровня. **Работает только в сети определенного типа(или ipv4 или ipv6)**

Есть версия HDLC от CISCO - **Он поддерживает мультипротокольную среду, не зависит от сетевого уровня**. Для этого помимо данных в нем находится поле protocol - тип протокола

CHDLC - открытый протокол, но чаще всего только на цисочных устройствах

FCS - контрольная сумма, используется для обнаружения ошибок при передаче

## Принцип работы PPP

![alt text](image-6.png)

PPP решает проблемы HDLC, связанные с работой в сетях одного типа и зависимость от вендора, т.к он мультивендорный и не зависит от типа сети


Содержит 3 компонента:
* Кадр (HDLC-Like Encapsulation)
* LCP - Link Control Protocol - отвечает за управление каналом передачи и позволяет устанавливать и настраивать соединение канала
* NCPs - Протокол управления сетями - используется для настройки протоколов сетевого уровня. Можно использовать несколько протоколов сетевого уровня

Есть поддержка аутентификации, а также отслеживание качества канала связи

PPP - протокол устаревший, но является предшественником современных протоколов канального уровня

### PPP Layered Architecture

![alt text](image-7.png)

PPP - имеет уровневую архитектуру

Физический уровень определяет среду передачи данных. Например Синхронные физические среды или асинхронные среды. **Используются только полнодуплексные последовательные каналы связи**

На канальном уровне читай со слайда. Он обеспечивает автоматическую настройку интерфейсов на концах, ограничения на размеры пакетов, обнаружение ошибок, следит за отключением каналов, качеством соединения каналов

После установки Канала связи PPP использует LCP для автоматического согласования форматов инкапсуляции и разных параметров (аутентификация, сжатие и обнаружение ошибок)

На сетевом уровне работает семейство протоколов NCP, с помощью которых возможна работа разных протоколов сетевого уровня модели OSI(ipv4 ipv6) на одном и том же канале связи. Для каждого сетевого протокола модели OSI используется своя реализация NCP протокола

Варианты NCP протоколов:
* IPCP - ipv4
* IPv6CP - ipv6

такие протоколы содержат поля со стандартизованными кодами, указывающими на протокол сетевого уровня. На каждый такой NCP протокол ложится ответственность инкапсуляции протоколов сетевого уровня в кадры PPP

Номера полей протокола PPP, которые используются в NCP, указывающие на инкапсулируемые протоколы
